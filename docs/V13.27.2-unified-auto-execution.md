# AlphaPilot V13.27.2 - Unified Automatic Execution

## Purpose

V13.27.2 gives OKX Demo and gated OKX Live the same operator workflow:

```text
start once -> scan closed candles -> arbitrate -> risk check ->
place protected order -> reconcile -> review -> continue
```

The workflow does not ask for per-order confirmation after an eligible runtime
has been started. Demo and Live share orchestration only; their credentials,
request semantics, adapters, releases, risk profiles, stores, ledgers, ARM
state, and emergency stops remain separate.

## Runtime model

- A backend runner owns the heartbeat; browser closure does not stop it.
- SQLite runtime access is serialized across the runner and HTTP threads.
- The heartbeat runs every 15 seconds for private-state reconciliation.
- A strategy is evaluated only when its timeframe has a newly closed candle.
- Checkpoints are scoped by environment, immutable release, and timeframe.
- Restarted processes retain desired state and checkpoints but never inherit an
  old process ARM. Credentials are process-only and must be supplied again.
- A failed preflight or reconciliation pauses new entries before any batch can
  submit an order.

## Demo workflow

Start Demo with Read + Trade credentials and both explicit gates:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\start_okx_demo_console.ps1 -EnableOrder -EnableAutomation
```

The credentials are shared by eligible Demo strategies for the current process,
but releases, positions, PnL, risk decisions, and evidence remain strategy
scoped. Demo private requests always include `x-simulated-trading: 1`.

## Live workflow

Live uses `scripts\start_okx_live_canary_console.ps1`. Automation requires the
master, read, Canary, order, and automation gates. The operator must also finish
read-only reconciliation and enter the current-process ARM phrase. Each Live
strategy still requires an approved immutable Live Release and matching active
RiskProfile before automatic execution can start.

Live requests never use the Demo header. Starting Demo can never arm Live, and
starting Live can never use Demo credentials or stores.

## Web and mobile controls

Both Demo and Live pages expose:

- `启动自动运行`
- `暂停新开仓`
- `停止`
- `紧急停止`
- enabled strategy count
- last heartbeat
- next evaluation
- latest result

The mobile status bridge is read-only. It reports backend automatic execution
state but cannot initiate an order or persist credentials.

## Hard safety boundary

- Raw API Key, Secret, and Passphrase are process-only.
- Withdraw, transfer, deposit, and API-key-management endpoints are absent.
- An immutable release and matching versioned RiskProfile are mandatory.
- Every order intent is idempotent and environment-scoped.
- Isolated margin and exchange-hosted TP/SL are mandatory.
- `rewardRiskRatio >= 2` remains a hard requirement.
- Live is off by default and requires a fresh process ARM.
- Automated tests use fake clients and never submit real exchange orders.

## HTTP surface

```text
GET  /api/auto-execution/runtime
POST /api/auto-execution/action
```

Supported actions are environment-scoped: start/arm, pause new entries, stop,
and emergency stop. The existing Live ARM endpoint also records the current
process ARM for the unified controller.

## Expected no-signal behavior

A completed scan with no matching strategy condition is a normal successful
heartbeat, not an error. The UI reports `等待策略条件匹配`; it does not invent a
signal or submit a placeholder order.
